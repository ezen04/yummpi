# 온보딩/진입 화면 추가 체크리스트

> 기준 화면(이미 작업됨): ① 랜딩 → ② 로그인/인증 → ③ 게스트 입장
> 근거 문서: `yummpi_prd_v2.1.md` · `api-spec.md` · `diagrams.md`(인증 플로우)
> 담당: ① 모임 라이프사이클 (온보딩·진입 FE)
> 상태 표기: ⬜ 미작업 / 🟨 부분 / ✅ 완료

---

## 우선순위 요약

| P | 그룹 | 이유 |
| --- | --- | --- |
| **P0** | 게스트 입장 에러/차단 · 입장 후 상태 라우팅 · 출발지 입력 | 없으면 흐름이 끊기거나 핵심 기능(중간지점) 불가 |
| **P1** | 카카오 로그인 중간 상태 · 진입 상태 분기 | 정상 동선은 돌지만 실패·예외 처리 공백 |
| **P2** | PWA 설치 안내 · 게스트→회원 전환 안내 | 완성도·리텐션 |

## 1. 게스트 입장 — 에러/차단 화면 (P0)

> 화면 ③은 "정상 입장"만 존재. 인증 플로우의 `모임 유효?` → 아니오 분기 화면이 전부 빠짐.

| ⬜ | 화면명 | 트리거 | 분기 조건 | 관련 API / 에러코드 |
| --- | --- | --- | --- | --- |
| ⬜ | 모임 만료 안내 | 초대 링크 진입 | `expiresAt` 경과 | `409 MEETING_EXPIRED` |
| ⬜ | 인원 초과 안내 | 입장하기 | `maxMembers` 초과 | `409 MEETING_CAPACITY_EXCEEDED` |
| ⬜ | 잘못된 초대 코드 | 링크/코드 진입 | 코드 없음·오타 | `400 INVALID_INVITE_CODE` / `404 MEETING_NOT_FOUND` |
| ⬜ | 종료된 모임 안내 | 초대 링크 진입 | 모임 `COMPLETED`/`CANCELLED` | (상태 기반 차단) |
| ⬜ | 닉네임 중복 (인라인) | 입장하기 | 동일 닉네임 존재 | `409 NICKNAME_DUPLICATED` → suffix 안내(지훈→지훈2) |

- 차단 화면 공통: 모임명 노출 없이 사유 + "새 모임 만들기/홈으로" CTA.
- 닉네임 중복은 별도 화면이 아닌 화면 ③ 인라인 에러 상태로 처리.

---

## 2. 입장 후 상태별 라우팅 (P0 · 화면보다 규칙)

> "입장하기"를 누르면 어디로 가나? 모임 `status`에 따라 진입점이 달라져야 함.

| ⬜ | 모임 상태 | 입장 후 도착 화면 | 비고 |
| --- | --- | --- | --- |
| ⬜ | `RECRUITING` | 대기/후보 등록 화면 | 투표 시작 전 |
| ⬜ | `VOTING` | 실시간 투표 화면 | 화면 ③ 배지 "장소 투표중" 케이스 |
| ⬜ | `PLACE_CONFIRMED` | 확정 장소·예약 안내 | |
| ⬜ | `IN_PROGRESS` | 참석 체크/진행 화면 | |
| ⬜ | `SETTLING` | 정산·소비 선택 화면 | |
| ⬜ | `COMPLETED` | 종료 안내 (1번 차단과 연결) | 입장 차단 |

- 이 매핑은 화면이 아니라 **라우팅 규칙**이므로 Figma보다 팀 합의 문서로 먼저 확정 권장.

## 3. 출발지 입력 + 위치 동의 (P0)

> 화면 ③은 닉네임만 받음. F2 중간지점 계산용 출발지 수집 단계가 흐름에 없음.

| ⬜ | 화면명 | 트리거 | 내용 | 관련 API |
| --- | --- | --- | --- | --- |
| ⬜ | 출발지 입력 | 입장 직후 or 투표 진입 시 | 주소/지하철역 입력 (좌표 변환) | `POST .../members` (`startAddress`·`startStation`·`startLatitude`·`startLongitude`) |
| ⬜ | 위치 수집 동의 고지 | 출발지 입력 진입 시 | 수집 목적·폐기 시점 안내(모임 종료 후 좌표 폐기) | (PRD 개인정보 원칙) |
| ⬜ | 출발지 미입력 안내 | 출발지 건너뜀 | 중간지점 계산 제외됨 표시 | (엣지케이스: 가중치 0) |

- **결정 필요**: 출발지를 ⓐ입장 직후 받을지 ⓑ투표/장소 단계에서 받을지 → 동선 합의.

## 4. 카카오 로그인 중간 상태 (P1)

> 회원 흐름에 OAuth 과정·실패·최초 가입 화면이 없음.

| ⬜ | 화면명 | 트리거 | 분기 조건 | 비고 |
| --- | --- | --- | --- | --- |
| ⬜ | 카카오 리다이렉트 로딩 | "카카오로 시작하기" | OAuth 진행 중 | 콜백 처리 스피너 |
| ⬜ | 로그인 실패/취소 | 카카오에서 취소·오류 | 인가 코드 미수령 | "다시 시도" CTA |
| ⬜ | 최초 가입 온보딩 | 신규 User upsert 직후 | 첫 로그인 | 닉네임 확정 + 이메일 선택 동의 |

## 5. 진입 상태 분기 (P1)

| ⬜ | 화면/규칙 | 트리거 | 분기 조건 | 처리 |
| --- | --- | --- | --- | --- |
| ⬜ | 로그인 회원 자동 리다이렉트 | 랜딩/로그인 방문 | 유효 세션 존재 | 바로 대시보드로 |
| ⬜ | 회원이 초대 링크 입장 | 회원이 `/join/:code` | 세션 보유 | 닉네임 입력 생략·세션으로 join (분기 정의) |
| ⬜ | 초대 코드 직접 입력 | 랜딩 "초대 링크로 입장하기" | 링크 없이 진입 | 코드 입력 화면 필요 여부 결정 |

- "초대 링크로 입장하기"가 항상 딥링크 전제인지, 코드 직접 입력도 허용인지 **결정 필요**.

## 결정 대기 항목 (팀 합의 필요)

1. 출발지 입력 시점 — 입장 직후 vs 장소/투표 단계
2. "초대 링크로 입장하기" — 딥링크 전용 vs 코드 직접 입력 허용
3. 회원의 초대 링크 입장 시 닉네임 처리 — 세션 닉네임 사용 vs 모임용 별도 닉네임
4. 입장 후 상태별 라우팅 규칙 확정 (2번 표)
