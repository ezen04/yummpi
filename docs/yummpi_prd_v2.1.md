## 1. 제품 개요

**제품명**: 얌피(yummpi)

**한 줄 소개**: 모임 장소 선정 → 투표 → (예약 관리) → 비용 정산을 하나의 서비스에서 처리하는 그룹 모임 운영 플랫폼.

**해결하려는 문제**: 모임 하나를 위해 카카오톡(날짜 논의) → 네이버 지도(식당 탐색) → 캐치테이블(예약) → 카카오톡(정산 요청) → 계산기(더치페이)까지 여러 앱을 오가야 한다. 장소 선정과 정산에서 특히 시간이 많이 든다. 얌피(yummpi)는 이 과정을 한 흐름으로 묶는다.

**앱 형태**: **Next.js 기반 PWA** (앱스토어 출시 없음, "앱처럼" 동작). 홈 화면 설치·전체화면·웹푸시 지원. 추후 필요 시 Capacitor로 스토어 출시 가능한 구조 유지.

## 2. 목표

**사용자 목표**: 모임 장소를 빠르게 결정한다 · 참석자 의견을 쉽게 모은다 · 정산을 확실히 끝낸다.

**비즈니스 목표**: 반복 모임(회식·동호회·스터디·가족) 사용자 확보 · 자체 모임/후기 데이터 축적 · 향후 예약·결제 플랫폼 연동.

**성공 지표(KPI)**:

- **모임 완주율**: 생성된 모임 중 정산 종료까지 도달한 비율 (목표 60%+)
- **정산 완결률**: 정산 시작 모임 중 전원 송금 완료 비율 (목표 70%+)
- **재사용률**: 모임 1회 이상 만든 사용자가 30일 내 추가 모임 생성 비율 (목표 30%+)
- 보조: 모임당 평균 참석자 수, 투표 참여율, 진입 대시보드 주요 액션 클릭률

## 5. 권한 모델

**인증 방식(확정)**: **카카오 소셜 로그인**(회원) + **게스트 닉네임 입장**(참석자). 자체 이메일/비밀번호 로그인은 사용하지 않음 → 비밀번호 해싱·재설정·이메일 인증 불필요. 소셜이 본인 인증을 대신하므로 별도 이메일 인증 단계 없음.

- 카카오 소셜 로그인 후 세션으로 관리.
- 게스트는 닉네임만으로 모임 범위 세션 생성.
- 향후 구글 등 소셜 로그인 추가 가능한 구조.

회원가입 허들을 낮추기 위해 **게스트(닉네임) 입장**을 허용하되, 책임이 따르는 행위는 회원으로 제한한다.

| 행위                          | 비회원(게스트 닉네임) | 회원        |
| ----------------------------- | --------------------- | ----------- |
| 초대 링크로 입장              | ✅                    | ✅          |
| 후보 장소 투표                | ✅ (닉네임 식별)      | ✅          |
| 후보 장소 추가                | ✅                    | ✅          |
| 모임 생성                     | ❌                    | ✅          |
| 예약 담당 지정·확정           | ❌                    | ✅ (주최자) |
| 정산 시작·확정                | ❌                    | ✅ (주최자) |
| 소비 항목 선택 / 송금 상태    | ✅ (닉네임 기준)      | ✅          |
| 진입 대시보드 / 내 모임 관리  | ❌                    | ✅          |
| 마이페이지 프로필 / 알림 설정 | ❌                    | ✅          |

**익명 투표 ↔︎ 실명 정산 충돌 해결**:

- 투표 익명 여부는 **모임 단위 설정**(`anonymousVoting`, 기본 익명)이다. 익명일 때 누가 무엇에 투표했는지 비공개하되, 내부적으로는 참석자(모임 내 고유 memberId)에 매핑한다. 투표 시작 후에는 설정 변경 불가.
- 정산/송금 현황은 **닉네임 단위로 실명화**한다. 즉 "지훈 18,000원 미송금"처럼 닉네임으로 식별. 게스트도 모임 내에서는 memberId로 일관 추적되므로 정산 대상 관리에 문제 없음.
- 같은 닉네임 중복 시 입장 순서 suffix(예: 지훈, 지훈2)로 구분.

## 6. 핵심 기능

### F0. 진입 대시보드 & 마이페이지 (회원 전용)

로그인 후 사용자가 가장 먼저 보는 화면은 **진입 대시보드**로 구성한다. 대시보드는 사용자의 모임 상태를 요약하고, 다음 행동으로 빠르게 이동하게 하는 역할을 한다.

- **진입 대시보드**: 오늘/예정 모임, 진행 중인 모임 상태, 투표 필요, 예약 필요, 정산/송금 필요 상태를 카드 형태로 요약.
- **빠른 액션**: 새 모임 만들기, 참여 중인 모임 이어가기, 송금 필요 모임 확인.
- **마이페이지**: 카카오 프로필 기반 닉네임 확인/수정, 내가 만든 모임/참여한 모임 구분 조회, 알림 설정, PWA 설치 상태 안내.

### F1. 모임 생성 (회원 전용)

입력: 모임명 · 날짜 · 인원 · 지역 · 예산(1인 기준) · 음식 종류 · 주차 여부 · 룸 여부 → 모임 + 초대 링크 생성.

- 예산은 장소 추천(F2)의 가격대 필터로 사용.

### F2. 장소 자동 추천

참석자 출발지 기반 **중간 지점을 먼저 계산**한 뒤, 해당 지점 주변을 **카카오 Local(키워드/카테고리) API**로 검색해 후보 장소 리스트를 생성한다. (예: 강남역 한식 / 고깃집 / 룸식당)

- **중간 지점 계산 (MVP)**: 참석자 출발지 좌표 기반 직선거리 최적 중간 지점 산출. 주최자가 "총 이동 최소(효율)" 또는 "최대 이동 최소(공평)" 기준 중 선택 가능. 출발지 미입력 참석자는 계산에서 제외.
- **예약 채널 플래그**: 검색 결과에 네이버예약/캐치테이블 링크가 있으면 "예약 가능" 뱃지 + 외부 딥링크 표시.

### F3. 실시간 투표

참석자가 후보 장소 중 하나에 투표하는 **단일 선택(1인 1표)** 방식. 투표 변경 가능(기존 표 교체), 실시간으로 전원 화면 동기화.
익명 여부는 모임 설정(`anonymousVoting`)을 따른다 — 기본 익명, 투표 시작 후 변경 불가. (식별은 권한 모델 참조)

### F4. 최종 장소 확정

투표 종료(마감 시각 or 주최자 수동) + 주최자 확정 → 최종 장소 결정.

### F5. 예약 담당 지정 & 상태 관리

주최자가 예약 담당자 지정. **앱은 예약 상태만 수동 관리**: 예약 전(NONE) → 진행 중(PENDING) → 완료(DONE). 상태 변경은 주최자 또는 예약 담당자.
(※ 실시간 빈자리 조회는 공개 API 부재로 미지원. 실제 예약은 외부 링크/전화로.)

### F6. 참석 체크

모임 시작 전 참석 여부 최종 확인 → 정산 대상 자동 생성.

### F7. 영수증 기반 자동 정산

영수증 촬영(PWA 카메라) 또는 갤러리 업로드 — **여러 장 일괄 접수** 가능 (호스트 전용) → **CLOVA General OCR**로 텍스트 추출 후 **자체 파서**로 메뉴 구조화(영수증별 독립 처리, 1장 실패해도 나머지 진행). 파서 상세: `ocr-parser-work-doc.md`.

- **OCR 결과 검수 단계(필수)**: 추출된 메뉴-가격을 사용자가 화면에서 확인·수정. OCR은 100% 정확하지 않으므로 검수 UI를 기본 흐름에 포함. 검수 화면 구성 요소:
  - 품목 목록 인라인 수정 + 신뢰도 0.8 미만 항목 하이라이트
  - **미분류 텍스트 영역**: 파서가 분류 못 한 줄을 사용자가 탭 → 이름/수량/금액 입력해 품목으로 승격
  - 검산 경고 배너: 합계 불일치 시 차액과 의심 라인 표시 (검수 통과 없이 소비 선택 진행 불가)
  - 합계/부가세/봉사료/할인 직접 수정 (봉사료는 차액 산출값임을 안내)
- **OCR 실패 fallback**: 인식 실패 시 1/N 균등분배 또는 금액 직접 입력 모드로 전환.

### F8. 참석자 소비 선택

각자 먹은 항목 선택(예: A=삼겹살·냉면 / B=삼겹살 / C=소주·삼겹살).

### F9. 자동 비용 분배

소비 항목 기반 자동 계산 + **봉사료/부가세 자동 배분**(메뉴 합계와 영수증 총액 차이 분배). 정산은 모임당 1건, 영수증 N장 합산.

### F10. 송금 현황 관리 & 완결

상태(미송금/송금 완료) 실시간 확인.

- **송금 딥링크**: 토스/카카오페이 송금 화면을 호출. ⚠️ **리스크**: 금액·계좌 prefill 가능 여부는 각 앱의 딥링크 스킴 정책에 종속 → F10 착수 전 ⑤가 실제 가능 범위 검증 필요. 불가 시 "금액 복사 + 앱 열기"로 후퇴.
- **송금 독촉 자동 알림**: 미송금자에게 D+1/D+3 자동 리마인더 발송. (iOS 미설치 대비 이메일 fallback)

```
카카오 로그인 → 진입 대시보드 → 모임 생성/내 모임 선택 → 장소 추천 → 실시간 투표 → 장소 확정 → 예약(상태 관리)
→ 모임 진행 → 참석 체크 → 영수증 업로드(OCR→검수) → 소비 선택 → 자동 정산

```

## 9. 데이터 모델

**기술 설계 문서의 ERD v2.2(`docs/erd-v2.2.md`)로 대체한다.** 이 문서에는 데이터 모델을 중복 기술하지 않는다 (구버전 표 삭제 — Gathering/Participant 모델은 폐기됨).

- 모임 상태: `DRAFT → RECRUITING → VOTING → PLACE_CONFIRMED → IN_PROGRESS → SETTLING → COMPLETED` (+ `CANCELLED`)
- 예약 상태: `NONE → PENDING → DONE`
- API 명세: 기술 설계 문서 v2.0 + `docs/api-v2.1.md` 변경분

## 10. 정책 · 엣지케이스

개발자별 상이한 구현을 막기 위한 사전 규칙.

- **투표 동률**: 동률 시 주최자가 최종 선택(런오프 없음, MVP 기준).
- **모임 취소/삭제**: 주최자만 가능. 정산 시작(COMPLETED 진입) 이후에는 삭제 불가, 보관 처리.
- **정산 확정 후 수정**: 소비 선택이 시작되면 영수증 구성은 잠긴다. 주최자가 영수증 선별 단계로 되돌아가면 참석자들의 소비 선택은 초기화되며, 이를 사전에 경고한다. 정산 확정(송금 단계 진입) 후에는 금액이 잠기며, 수정하려면 주최자가 정산을 "재오픈"해야 함(전원 미송금 상태일 때만 — 송금 완료 1건 이상이면 재오픈 불가).
- **영수증 다중**: 한 모임에 영수증 N장 허용(2차, 추가 주문 등). 정산은 모임당 1건으로 영수증 합산.
- **참석자 0~1명**: 투표/정산 스킵 가능. 1명이면 정산 = 전액 본인.
- **출발지 미입력 참석자**: 최적장소 계산에서 제외(가중치 0), 안내 표시.
- **링크 만료/모임 종료 후 입장**: 입장 차단, 종료 안내 화면.
- **대시보드 데이터 범위**: 로그인한 회원의 주최 모임과 회원 계정으로 연결된 참여 모임만 표시한다. 게스트 닉네임으로만 입장한 모임은 계정에 자동 귀속하지 않는다.
- **마이페이지 알림 설정**: 웹푸시 권한이 거부되었거나 PWA 미설치 상태인 경우, 설정 화면에서 현재 상태와 해결 방법을 안내한다.

## 11. 개인정보 처리

- **출발지 위치**: 최적장소 계산 목적에 한해 수집, 입력 시 동의 고지. 모임 종료 후 좌표 폐기(집계용 비식별 데이터만 보관).
- **영수증 이미지**: S3 저장, 보관 기간 한정(예: 모임 종료 후 90일 후 삭제). 카드번호 등 민감정보는 OCR 단계에서 마스킹 처리.
- **카카오 프로필**: 로그인 시 카카오 계정 ID·닉네임을 수집(이메일은 선택 동의). 최소 수집 원칙 — 프로필 이미지 등 불필요 항목은 동의 요청하지 않음.
- **마이페이지 정보**: 닉네임, 알림 수신 여부, PWA 설치 상태처럼 서비스 사용에 필요한 최소 설정 정보만 다룬다.
- **게스트 닉네임**: 모임 범위 내에서만 유효, 외부 노출 없음.
- 결제/송금은 외부 앱(토스/카카오페이)에서 이뤄지므로 **계좌·결제 정보는 앱에 저장하지 않음**.

## 12. 비기능 요구사항 (NFR · MVP 가정)

- **모임 규모**: 한 모임 최대 참석자 30명 가정(실시간 투표 동시성 기준).
- **동시성**: 동시 활성 모임 수백 건 수준에서 실시간 서버 수평 확장 가능 구조.
- **응답시간**: 일반 API p95 < 500ms, 장소검색(외부 API 포함) < 1.5s 목표.
- **가용성**: MVP 99% (단일 리전, 무중단 배포는 V2).
