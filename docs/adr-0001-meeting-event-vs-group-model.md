# ADR-0001 — 모임 모델: 단일 이벤트(A) vs 지속 그룹(B)

| 항목 | 내용 |
| --- | --- |
| 상태 | **결정됨 (2026-06-18)** — 팀 만장일치: A + B-lite 클론 + §6 위생 / B는 V2 |
| 작성일 | 2026-06-18 |
| 작성 | ① 모임 라이프사이클·Prisma 스키마 오너 |
| 영향 | 전 도메인 (①②③④⑤) |
| 관련 문서 | `docs/erd-v2.2.md` · `docs/api-spec.md` · 루트 `CLAUDE.md` |
| 권고 결론 | (확정) **순수 A 유지 + B-lite "이 멤버로 새 모임 만들기" 클론 + §6 전방호환 위생. 지속 그룹(B)은 V2. `series_id` 스탬프만 지금 도입.** |

---

## 1. 배경 / 문제 제기

ERD 검토 중, 현재 설계가 **모임 = 단일 이벤트**를 전제함을 확인했다 (`settlements.meeting_id UNIQUE` — 모임당 정산 1건). 얌피의 핵심 가치가 "하나의 모임을 만들어 **지속 관리**"라면, 같은 멤버가 다시 만날 때마다 새 모임을 만들어야 하는 구조는 "모임 관리 앱"이 아니라 "이벤트 기록 앱"에 가깝다.

이 전제 차이는 ERD·상태머신·정산·송금 전반에 영향을 주므로, 구현이 더 진행되기 전에 결정을 명문화한다.

---

## 2. 핵심 관찰 (결정의 전제)

### 2-1. 현재 ERD는 이미 B에 절반쯤 가깝다
- `meeting_members` · `invite_code` · `host`(멤버십·초대·소유)는 **이미 이벤트 흐름과 별도 테이블**로 분리돼 있다.
- A안이 한 일은 "새 계층을 안 만든 것"이 아니라, 투표·장소·예약·정산·송금이라는 *이벤트 흐름*을 Meeting 행에 **융합**시켜 둔 것이다.
- 따라서 B로 가는 작업 = 새 레이어 추가가 아니라, **Meeting 한 테이블의 책임을 Gathering으로 분리(lift)**. 체감보다 간격이 좁다.

### 2-2. 게스트 설계(b안)가 B의 상한을 이미 깎아놓았다
- 게스트는 b안 → `users` 미생성 · 모임 범위 서명 토큰. 구조적으로 **다음 모임에 따라올 수 없다.**
- B의 핵심 가치("멤버 재초대 불필요 + 그룹 히스토리")는 **카카오 회원에게만** 적용된다.
- 반복 멤버 다수가 게스트라면 B의 ROI는 예상보다 작다.

### 2-3. A안 = "Meeting당 Gathering 1개"인 B안의 특수 케이스
- 그래서 A→B 마이그레이션은 **재작성이 아니라 기계적 리프트**다 (§7 정밀 비용 참조).

### 2-4. 결정적 타이밍 — ②④⑤가 아직 미구현
- 코드 산정(2026-06-18) 결과 **장소·정산·송금 API는 미구현**(features/*/CLAUDE.md 플레이스홀더만 존재).
- B 영향이 가장 큰 두 도메인(④ 정산·⑤ 송금)이 아직 `meeting_id`에 코드를 안 박았다.
- → **데이터 모델 전환 비용이 ①에 집중되고, 지금이 역사상 가장 싼 시점.** ④⑤가 작성을 시작하면 이 이점이 사라진다.

## 3. 옵션

### A안 — 단일 이벤트 모델 (현행)
모임 = 단일 약속. 종료 후 닫힘. 재회동 시 새 모임 생성.

```
DRAFT → RECRUITING → VOTING → PLACE_CONFIRMED → IN_PROGRESS → SETTLING → COMPLETED
```

- 정산: `settlements.meeting_id UNIQUE` — 모임당 1건
- 장점: 상태머신 선형·단순. ERD 변경 없음. 핵심 루프 검증에 충분.
- 단점: 매번 재초대(회원). 그룹 히스토리 없음. V2 전환 시 라이브 데이터 마이그레이션 리스크.

### B안 — 지속 그룹 모델 (full)
모임 = 지속 그룹. 하위에 약속(Gathering)이 N회 반복. 스키마 + 신규 화면 전부 지금 반영.

```
모임 (Meeting) — 멤버 고정, 지속
  └─ 약속 (Gathering) ×N
        └─ 투표 → 장소확정 → 예약 → 정산 → 송금
```

- 장점: 반복 모임 완전 지원. 앱 정체성과 일치.
- 단점: 신규 화면(그룹홈·약속목록·약속생성)까지 MVP에 포함 → **25일 일정 위협.** 전 도메인 동시 영향.

### B-lite — 저비용 절충 (ERD 변경 없음)
- **"이 멤버로 새 모임 만들기" 클론 액션** — 종료 화면에서 이전 모임의 *회원* 멤버를 복사해 새 Meeting 생성. 순수 UX/카피 → 재초대 페인 해소.
- (선택) `series_id` self-FK(nullable)로 히스토리 묶기. 이벤트 흐름은 Meeting에 그대로.

### 제3안 — "B 기반(schema) / A 경험(UX)" 하이브리드 ★ 권고 후보
스키마는 B로 **지금** 깔되, MVP 경험은 A로 출시.
- 모임 생성 시 **Gathering 1개 자동 생성**, UI는 약속 개념을 숨김.
- 이벤트 흐름 FK는 처음부터 `gathering_id`. ④⑤는 그 위에 신규 작성.
- 장점: **V2가 "위험한 데이터 마이그레이션 + 5인 리팩터"에서 "순수 기능 추가"로 바뀜.** ④⑤ 미래 재작업 0.
- 단점: MVP 내내 안 보이는 개념(Gathering) 때문에 테이블 1개 + 조인 증가(복잡도세). **비용이 ①에 집중**(이미 최다 부하 — 팀원1 지적).

## 4. 결정 (권고)

> **제3안(B 기반 / A 경험)을 1순위로 검토한다.** ①·③의 *이미 구현된* 코드 재작업 규모(§7)가 **~2~4일 안**이면 채택한다. 그보다 크게 나오면 **순수 A + 전방호환 위생(§6)**으로 후퇴하고 V2 데이터 마이그레이션을 감수한다.
>
> **신규 화면(그룹홈·약속목록 등)은 어느 경우든 MVP 제외, V2.** 재초대 페인은 **B-lite 클론 액션**으로 MVP에서 해소한다.
>
> ⏰ **이 결정은 ④⑤가 정산·송금 코드를 쓰기 시작하기 전에 내려야 가치가 있다.** 그 후엔 제3안의 이점이 소멸하고 순수 V2 마이그레이션만 남는다.

### 근거
1. MVP 목적은 **핵심 루프(투표→정산→송금) 검증**이며, 단일 이벤트 모델로 100% 증명된다.
2. 데이터 모델 비용은 ②④⑤ 미구현 덕에 **지금이 가장 싸다**(§2-4).
3. 게스트 설계가 B 상한을 이미 깎았다(§2-2) → 풀 B(신규 화면 포함)는 ROI 대비 일정 리스크 과다.
4. 팀 의견 수렴: 제안자·팀원 2인 모두 "B 지향, 풀 전환은 V2"로 일치(§9). 쟁점은 "지금 B를 얼마나 깔 것인가"로 좁혀졌다.

## 6. 순수 A 후퇴 시 전방호환 위생 (지금 ①이 보장)

제3안을 안 가더라도 V2 리프트를 싸게 유지하려면:

- [ ] `settlements.meeting_id UNIQUE` 유지("이벤트 단위당 정산 1건" 문서화 헤지).
- [ ] 소켓 핸들러를 단일 id 변수로 파라미터화(③) → room 키 교체가 1줄.
- [ ] 정산/송금 합계를 멤버 행에 "정산 1건 가정"으로 비정규화 금지(④⑤).
- [ ] 이벤트 의미(투표·장소·예약·정산·송금)와 그룹 의미(멤버·초대·호스트)를 코드에서 격리.

## 7. 정밀 비용 산정 (코드 기준 · 2026-06-18)

### 7-1. 스키마 — 이벤트 흐름을 Gathering으로 이전

`gatherings` 테이블 신설 후, **직접 `meeting_id` FK 5개**를 `gathering_id`로 리포인트:

| 모델 | 현재 | 비고 |
| --- | --- | --- |
| `PlaceCandidate` | `meeting_id` + `@@unique([meetingId, externalPlaceId])` + `confirmedForMeeting` | → gathering |
| `Vote` | `meeting_id` + `@@unique([meetingId, memberId])` (1인1표) | → gathering |
| `Reservation` | `meeting_id @unique` (모임당 1예약) | → gathering |
| `Receipt` | `meeting_id` | → gathering |
| `Settlement` | `meeting_id @unique` (모임당 1정산) | → gathering |
| `Payment` | `settlement_member_id`만 보유 | **직접 FK 없음 → 자동 따라감(추가비용 0)** |

`Meeting`에서 Gathering으로 이전: `status` · `voting_closes_at` · `confirmedCandidate`.
**`MeetingMember`는 Meeting에 잔류**(멤버십=그룹 레벨). `invite_code` · `host`도 잔류. → 분리선이 깨끗하다.

### 7-2. 이미 구현된 코드 재작업 (① 집중)

| 파일 | 작업 | 규모 |
| --- | --- | --- |
| `lib/meeting-status.ts` | 로직 동일, `meetingId`→`gatheringId` 파라미터·쿼리 대상 변경 | 小(기계적) |
| `api/v1/meetings/route.ts` | 생성 시 Meeting + Gathering 1개 자동 생성, 목록 status 필터 이전 | 中 |
| `api/v1/meetings/[meetingId]/route.ts` | `meeting.status`·`VOTING_ENTERED`·`canTransition(CANCELLED)` 읽기를 gathering 기준으로 | 中 |
| `api/v1/meetings/[meetingId]/status/route.ts` | PATCH 대상 gathering | 小 |
| `api/v1/meetings/invite/[inviteCode]/route.ts` | `meeting.status` 반환 → 그룹+현재 약속 형태로 | 小 |

### 7-3. ③ 소켓 (apps/server/handlers/meetingHandlers.ts)
- room 키 `meeting:${meetingId}` **4곳** + `member:joined/left` 페이로드 → `gathering:` 컨벤션. **小(1파일, 수 줄).**
- 투표 영속(`vote:cast`) 핸들러는 **미구현** → 처음부터 gathering FK로 작성(추가비용 0).

### 7-4. ②④⑤ — **추가비용 ≈ 0**
- 장소·정산·송금 API 미구현(플레이스홀더만). 처음부터 `gathering_id`로 작성하면 끝.

### 7-5. 합산
- 제3안 비용 ≈ **① 스키마 1마이그레이션 + 위 5개 라우트 재작업(~2~3일) + ③ room 키(~수 줄)** + 신중한 검증.
- 풀 B(지금) = 위 + **신규 화면(그룹홈·약속목록·약속생성)** → 일정 위협 → **MVP 제외**.
- V2로 미룰 때(④⑤가 meeting_id로 완성된 뒤) = 위 전부 + **④ 정산엔진·⑤ 송금 재작성 + 라이브 데이터 마이그레이션.** → 가장 비쌈.

> 결론: 데이터 모델 비용은 **지금 ① 집중 2~4일**이 바닥값. 단 ①이 최다 부하(라이프사이클+스키마+상태머신) 상태라 **부하 분산**(③ room 키, ⑤ packages/schemas 반영 등)을 페어링으로 가져가야 함.

## 8. 논의 요청 / 미해결 질문 (#dev — 이번 주 1순위)

1. **제3안 채택 여부**: ①·③ 재작업이 §7 추정(~2~4일) 내인지 ①이 실측 → 채택/후퇴 결정.
2. **신규 화면**: 그룹홈·약속목록은 V2 확정(이견 없음).
3. **B-lite 클론 액션**: MVP(P1?) 포함 여부.
4. **⑤ 송금 착수 일정**: 제3안 채택 시, ④⑤가 코드 쓰기 전 스키마 확정 선행 필요 → 일정 동기화.
5. **전방호환 위생(§6)**: 후퇴 시 ③④⑤ 합의.
